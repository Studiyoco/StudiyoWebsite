04 The Rive workflow

How a mascot
gets built.

This is the exact sequence we run, with the real day counts from a recent full character system. Each phase gates the next, so nothing gets animated before the thing underneath it is solid.

P1

Designing the character

Two visual directions, you choose one. Then turnarounds at every angle it appears, and all expressions sketched flat. This sets the creative direction for everything after it.

3
days
P2

Building the animation system

The engine. The rig that lets the character look in any direction and show any emotion at any moment, tested at every size. Invisible to users, but everything else depends on it.

3.5
days
P3

Expressions and emotions

Every expression animated, plus ambient mood loops. Because emotions sit on their own layer, each one works at any head position in any animation with no extra production.

7
days
P4

In-app moments

Every key moment a user hits, each mapped to a trigger. The wait states matter most: when the AI is working, the mascot fills the silence and the product feels alive.

6.5
days
P5

Celebrations and props

The milestone moments users remember, longer and more expressive, plus the props that make the character belong in your product. Designed to feel special and shareable.

5.5
days
P6

Polish and handoff

Every loop checked, every emotion tested at every size. Your team gets the final files plus a plain-language guide for triggering each state from code. No technical background needed to follow it.

4
days

Total for a full character system

≈ 29.5 working days · 4 to 7 weeks elapsed

05 The state machine

From a sketch
to a system.

Before we build anything in Rive, we map every state and how the character moves between them. Here is an actual planning sketch, and the clean machine it becomes. Click a trigger to run it.

Planning artifactstate-machine.sketch
Rive · Layer 1 · characterstate: IDLE
how your developers use itonboarding.tsx
// one file, named inputs. no frame math, no asset juggling.
const machine = rive.stateMachine("character");

// user got it right
machine.fireInput("correctAnswer");   // IDLE -> CORRECT -> NEUTRAL

// user got it wrong
machine.fireInput("incorrectAnswer"); // IDLE -> INCORRECT -> NEUTRAL

// back to resting
machine.fireInput("reset");          // NEUTRAL -> IDLE
06 Why Rive

One file. Every
screen. Real logic.

Rive is not a video format and not a GIF exporter. It is a runtime. The animation and its logic ship together in a single vector file that your app drives in real time.

1

One rig, infinite combinations

Add an expression once and it works across every existing animation. Add an animation once and it works with every existing expression. Everything compounds.

~KB

Vector, not frames

A whole character system is a small vector file, not a folder of heavy exports. It stays crisp from a nav button to a full-screen moment.

4

Runs everywhere

The same file runs on iOS, Android, web, React Native, and Flutter through official runtimes. Build once, ship to every platform.

</>

Triggered from code

Named states and inputs your developers call directly. The character reacts to real events in your product, not a timeline that plays regardless.
